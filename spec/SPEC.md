# Portable WezTerm + Claude Workspace — Specification

**Goal:** A fully self-contained, portable WezTerm setup that can be copied to any Windows machine and run without a system install.
---

## 1. Directory Layout

```
Wezterm\
├── SPEC.md                   ← this file
├── bin\
│   └── wezterm.exe           ← portable WezTerm binary (downloaded by install.ps1)
├── config\
│   └── wezterm.lua           ← WezTerm config
├── scripts\
│   ├── install.ps1           ← one-time setup: downloads bin, registers context menu
│   ├── uninstall.ps1         ← removes context menu registry entries
│   └── launch-claude.ps1     ← called by context menu; opens WezTerm with 5 Claude tabs
└── README.md                 ← quick-start instructions (generated by install.ps1)
```

---

## 2. Portability Approach

- **No system install required.** The single `wezterm.exe` binary is self-contained.
- WezTerm distributes a portable ZIP on GitHub Releases — `install.ps1` downloads and extracts it.
- The config file is specified via the `WEZTERM_CONFIG_FILE` environment variable when launching, so it never touches `%APPDATA%` or the user profile.
- The only machine-specific side effect is two registry keys (context menu) — `install.ps1` writes them, `uninstall.ps1` removes them.
- To move to another machine: copy this entire folder, run `install.ps1` once.

---

## 3. `install.ps1` — One-Time Setup

**Responsibilities:**

1. **Download WezTerm portable binary**
   - Fetch the latest stable release ZIP from `https://github.com/wez/wezterm/releases/latest`
   - Filter for the asset matching `WezTerm-windows-*-portable.zip` (no installer, no MSI)
   - Extract only `wezterm.exe` (and `wezterm-gui.exe` / supporting DLLs if present) into `.\bin\`
   - Skip download if `.\bin\wezterm.exe` already exists (idempotent)

2. **Register right-click context menu entries** (see Section 5)

3. **Print confirmation** with paths, version installed, and how to uninstall

**No elevation required** for the binary download. Registry writes go to `HKCU` (current user), not `HKLM`, so no admin rights are needed.

```lua
config.initial_cols = 120

config.default_prog = { 'powershell.exe' }
config.window_close_confirmation = 'NeverPrompt'

  'claude.cmd', 'claude.exe', 'claude', 'node.exe', 'node'
}
```

Note: `config.default_cwd` is intentionally omitted — the launch script sets CWD per invocation.

### 4.2 Shift+Enter for multiline Claude prompts

Enable the Kitty keyboard protocol so WezTerm passes Shift+Enter as a distinct key sequence (separate from plain Enter), which Claude CLI uses for multiline input:

```lua
config.enable_kitty_keyboard = true
```

No explicit key binding is needed — with the Kitty protocol active, Claude CLI's readline interface receives the correct escape sequence (`\x1b[13;2u`) when Shift+Enter is pressed.

### 4.3 Launch menu (preserve + extend)

## 5. Right-Click Context Menu

### Entry point

Two registry keys are needed so the option appears both when right-clicking a **folder** and when right-clicking the **background** of an open folder:

```
HKCU\Software\Classes\Directory\shell\OpenWezTermClaude
HKCU\Software\Classes\Directory\Background\shell\OpenWezTermClaude
```

### Menu item label

```
Open Claude Workspace Here (5 tabs)
```

### Command value

```
powershell.exe -WindowStyle Hidden -ExecutionPolicy Bypass -File "C:\Users\TimDeBoer\Code\Wezterm\scripts\launch-claude.ps1" -TargetDir "%V"
```

For the `Directory\Background` key, `%V` resolves to the current folder.
For the `Directory` key, `%1` is the selected folder path — the command uses `%1` instead.

`install.ps1` writes both keys. `uninstall.ps1` removes them.

## 6. `launch-claude.ps1` — 5-Tab Claude Launcher

**Parameter:** `-TargetDir <path>` (the directory that was right-clicked)

**Tab layout:**

| Tab | Label | Model flag | Notes |
|-----|-------|-----------|-------|
| 1 | Sonnet 1 | `--model claude-sonnet-4-6` | First/active tab |
| 2 | Sonnet 2 | `--model claude-sonnet-4-6` | |
| 3 | Sonnet 3 | `--model claude-sonnet-4-6` | |
| 4 | Haiku    | `--model claude-haiku-4-5-20251001` | |
| 5 | Opus     | `--model claude-opus-4-6` | |

All tabs use `--dangerously-skip-permissions` and start with CWD = `$TargetDir`.

**Window title format:**

```
Claude - <TargetDir>
```

e.g. `Claude - C:\Users\SquishyBeefCake\Projects\MyApp`

**Launch sequence:**

```
Step 1 — Launch WezTerm with Tab 1 (Sonnet 1):
  Set WEZTERM_CONFIG_FILE=<this repo>\config\wezterm.lua
  Run: wezterm.exe start --cwd <TargetDir> -- claude --dangerously-skip-permissions --model claude-sonnet-4-6

Step 2 — Wait ~1 second for WezTerm to initialize and expose its IPC socket.

Step 3 — Set window title:
  Pane ID is returned by Step 1 (wezterm start prints the pane ID to stdout).
  Resolve the window ID: wezterm.exe cli list --format json | parse to find window ID for that pane.
  Run: wezterm.exe cli set-window-title --window-id <windowId> "Claude - <TargetDir>"

Step 4 — Spawn remaining tabs via wezterm CLI:
  wezterm.exe cli spawn --cwd <TargetDir> -- claude --dangerously-skip-permissions --model claude-sonnet-4-6
  wezterm.exe cli spawn --cwd <TargetDir> -- claude --dangerously-skip-permissions --model claude-sonnet-4-6
  wezterm.exe cli spawn --cwd <TargetDir> -- claude --dangerously-skip-permissions --model claude-haiku-4-5-20251001
  wezterm.exe cli spawn --cwd <TargetDir> -- claude --dangerously-skip-permissions --model claude-opus-4-6

Step 5 — Set tab titles (cosmetic):
  wezterm.exe cli set-tab-title --tab-id <id> "Sonnet 1" (etc.)
```

**Title persistence:** WezTerm will attempt to override the window title with the active pane's shell title (set via OSC 2 sequences) unless suppressed. To prevent this, set in `config/wezterm.lua`:

```lua
-- Prevent shell/process from overriding window title set by the launcher script
config.window_title_font_size = nil  -- no change needed here
-- The title override is blocked by disabling OSC title setting:
config.window_frame = { title_font_size = 12.0 }  -- cosmetic only
```

More reliably, the script should also set the title *after* all tabs are spawned (Step 5+), since Claude CLI may emit its own OSC title sequences on startup. A short delay + re-set is the safest approach.

Alternatively, `config/wezterm.lua` can define a `format-window-title` event handler that always returns a fixed format — but this would apply to all WezTerm windows, not just ones launched from the context menu. The script-based approach is preferred for isolation.

The script resolves the WezTerm socket automatically via `WEZTERM_UNIX_SOCKET` / `WEZTERM_PANE` environment variables that `wezterm cli` discovers from the running instance.

**WEZTERM_CONFIG_FILE** is set in the script before launch so the portable config is used regardless of what (if anything) is installed system-wide.

---

## 7. `uninstall.ps1`

Removes both registry keys written by `install.ps1`. Does not touch `bin\` or `config\`.

---

## 8. Machine Portability Checklist

| Requirement | How met |
|-------------|---------|
| No system WezTerm install | `bin\wezterm.exe` is the standalone portable binary |
| No admin rights | All registry writes use `HKCU`; binary is user-space |
| Config isolated from system | Launched with `WEZTERM_CONFIG_FILE` env var |
| Works on any machine with the folder | Copy folder → run `install.ps1` |
| Claude CLI assumed present | Claude is installed globally via `npm i -g @anthropic-ai/claude-code`; if absent, `install.ps1` should detect and warn |
| Sync via OneDrive / network share | Folder is plain files; only `bin\wezterm.exe` is large (~15 MB) |

---

## 9. File Implementation Order

1. `scripts/install.ps1` — download binary, write registry keys
2. `config/wezterm.lua` — based on existing config, with kitty keyboard + model launch menu
3. `scripts/launch-claude.ps1` — multi-tab Claude launcher
4. `scripts/uninstall.ps1` — registry cleanup

## 10. Open Questions / Decisions Needed

- **WezTerm version pinning:** Should `install.ps1` always fetch `latest`, or pin to a specific tag (e.g., `20240203-110809-5046fc22`)? Latest is convenient but risks breakage.
- **Claude CLI check:** Should `install.ps1` verify `claude` is on PATH and warn if not found?
- **Tab titles:** WezTerm's `cli set-tab-title` requires knowing the tab ID returned by `cli spawn`. The script will need to parse that output. If tab titles are not critical, this step can be omitted.
- **Shift+Enter fallback:** If a Claude CLI version older than the multiline-supporting release is on PATH, Shift+Enter will do nothing extra. No config change needed — just a user awareness note.
